# PRINCIPLE: AI-Driven Agent Platform

## 1. 核心的論理構造
本プロジェクトは、汎用AIエージェント基盤とオーケストレータを中核とし、AIによる完全自動化・クラウドコストの極限化・人間介入0%の「AI-Only Operations」を最上位目的とする。ニュースエージェントは本基盤上の最初のリファレンス実装である。

## 2. 実行強制命令(10ステップ・ワークフロー)
すべての思考・設計・出力は、以下10手順を一切の省略なく実行しなければならない。

1. [PLAN]: 最小3ステップの計画、検証方法、失敗予見と対策。
2. [SUBTASKS]: 並列/分割可能な単位への分解。
3. [LESSONS]: 過去ミス(省略、誤解、バグ、設計乖離)のパターン分析と防止策。
4. [PROGRESS]: [ ] / [x] による進捗可視化。
5. [EXECUTION]: コード・設計・設定の具体的出力。
6. [VERIFICATION & ALIGNMENT]: 期待と出力の比較に加え、計画および `SYSTEM_DESIGN.md` との齟齬を厳格検証。乖離時は Step 5 へ差し戻し。
7. [ELEGANCE CHECK]: より簡潔・高度・低コストな代替案の検討。
8. [PRINCIPLES CHECK]: 簡潔さ、根本原因解決、影響最小化の検証。
9. [PRINCIPLES EXPANSION CHECK]: 目標達成度、観測可能性、仮説検証、安全性、コスト、一貫性の確認。
10. [FINAL ANSWER]: そのターンの最終成果物の出力。

## 3. 不変性と完全性の原則
- **省略の禁止**: 「...」や「以下略」を一切禁じる。
- **無断削除の禁止**: ユーザー承認なしに既存成果物を削除・変更しない。
- **引き継ぎの義務**: セッション終了時に必ず `HANDOVER.md` を更新。
- **整合性の義務**: 設計(MD)と実装(Code)に1%の齟齬も許容しない。

## 4. 汎用性とプラガビリティの原則
- **基盤とアプリの分離**: `src/orchestrator/` と `src/agent/` は特定ユースケースに依存してはならない。用途固有ロジックは必ず `src/apps/<n>/` 配下に隔離する。
- **契約駆動**: ステージ間の入出力は `src/agent/contracts.py` のスキーマに従う。スキーマ変更はドキュメント同期を伴う。
- **宣言的定義**: ジョブとパイプラインは YAML による宣言的定義を唯一の正とし、コードへのハードコーディングを禁ずる。
- **部品の単一責務**: 各ステージは入力→出力の単一変換のみを担い、副作用はオーケストレータ経由で行う。

## 5. 開発哲学
- **Documentation First**: 実装前に設計書を更新し論理的整合性を担保。
- **Zero Manual Coding**: 人間によるコード編集を1行も許容しない。
- **Full Asset Delivery**: 成果物は常にZIP化可能な完全ディレクトリ構造で提示。
- **No Standing Server**: コスト最小化のため常駐プロセスを持たず、オーケストレータは都度起動とする。
